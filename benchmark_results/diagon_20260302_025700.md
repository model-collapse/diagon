## 5. Performance Analysis

### Strengths
- **Massive query speedup from single-segment index**: forceMerge(1) eliminated inter-segment overhead. OR-50 improved 60%, OR-20 improved 49%.
- **Index size reduced 40%**: 685 → 412 B/doc from freq-in-docDelta packing + delta FPs + merging
- **WAND scaling excellent**: 86x speedup at 50 terms shows Block-Max WAND is highly effective
- **All P99 targets exceeded by 7-114x margin**: Every query category well within targets
- **1605/1605 tests pass**: Full CI green

### Key Insight
The combined effect of Issue #8 optimizations is significant:
1. **forceMerge(1)**: Single segment eliminates 7-segment scan overhead → 20-60% faster queries
2. **freq-in-docDelta packing**: 32% smaller .doc → faster postings decode
3. **Delta-encoded FPs**: 52% smaller .tim → faster term dictionary traversal
4. **SKIP_INTERVAL 128**: Halved skip overhead → minor benefit

These compound to a **36% average P50 improvement** across all query types.

## 6. Issues and Concerns

1. **Throughput number misleading**: forceMerge(1) was added for fair index size comparison. The 5,173 docs/sec includes ~3.1s of merge time. Raw indexing speed is ~40,000+ docs/sec (unchanged).

2. **Cold-start P99 outlier**: First few queries hit cold pages (single-term oil P99: 1.241 ms). Steady-state is < 0.15 ms.

3. **Index still 1.37x Lucene**: Remaining gap is StreamVByte vs PFOR (.doc), position encoding (.pos), and FST prefix sharing (.tim).

## 7. Recommendations

1. **Separate forceMerge from throughput measurement**: Time the merge separately so indexing throughput remains comparable to baseline.
2. **Run Lucene comparison benchmark** (`/benchmark_reuters_lucene`) to validate speedup ratios after the encoding changes.
3. **Consider PFOR encoding** for .doc files to close the remaining 38% of the Lucene size gap.
